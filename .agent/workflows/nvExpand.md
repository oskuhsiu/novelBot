# /nvExpand - 章節擴寫

將章節擴寫至目標字數範圍。適用於：
- `/nvDraft` 產出的情節骨架 → 擴寫為完整正文
- 任何字數不達標的章節 → 補足至目標

> [!CAUTION]
> **絕對禁令**
> 1. **字數不達標 = 未完成 = 禁止進入下一步**。config 中的 `words_per_chapter.min` 是硬性下限，不可忽略。
> 2. **禁止中途停止**。一旦開始執行，必須完成所有指定的 `n` 章後才能停止。
> 3. **禁止一次思考所有章節**。必須一章一章地完成，每章都是獨立的完整任務。

## 參數

| 參數 | 必填 | 說明 | 範例 |
|------|------|------|------|
| `proj` | ✅ | 專案名稱 | `proj=霓虹劍仙` |
| `chapter` | ✅ | 起始章節號碼 | `chapter=5` |
| `n` | ❌ | 批次擴寫章數（預設 1） | `n=3` |

## 使用範例

```
/nvExpand proj=霓虹劍仙 chapter=5
/nvExpand proj=霓虹劍仙 chapter=5 n=3  # 連續擴寫 5, 6, 7 章
```

搭配 `/nvDraft` 使用：
```
/nvDraft proj=霓虹劍仙 chapter=5    # 先產出骨架（覆蓋到 chapter 檔案）
# （可手動編輯 brief 調整劇情方向）
/nvExpand proj=霓虹劍仙 chapter=5   # 再擴寫為完整正文
```

---

## 核心設計原則：逐章獨立處理

> [!IMPORTANT]
> **單章聚焦模式（One-Chapter-at-a-Time）**
>
> 本 workflow 的核心哲學是：**每次只思考、只處理一章**。
>
> ```
> ❌ 錯誤：一次載入所有 n 章的需求 → 同時規劃 → 批次產出
> ✅ 正確：載入第 i 章 → 完整擴寫第 i 章 → 儲存 → 清除上下文 → 載入第 i+1 章 → ...
> ```
>
> **理由**：LLM 在處理大量文字任務時，若同時顧及多章需求，會出現「偷懶」行為——
> 每章字數縮水、細節粗糙、描寫趨於概括化。逐章處理可確保每章都獲得 100% 的注意力。

## 執行步驟

### Step 0: 載入專案設定（僅一次）
// turbo
讀取並記錄以下設定，整個批次期間不會改變：

```yaml
必讀設定:
  config/novel_config.yaml:
    - words_per_chapter.min   # ← 硬性下限，不可忽略
    - words_per_chapter.max   # ← 硬性上限
    - words_per_chapter.target # ← 理想字數
    - content_weights          # ← 內容比例
    - style_profile           # ← 風格設定
  config/narrative_progress.yaml:
    - 當前進度
    - completed_chapters[]
  config/character_db.yaml:
    - 涉及角色狀態
  output/style_guide.md:
    - 風格指南（若存在）
```

顯示字數要求確認：
```
═══════════════════════════════════════════════════════
  ⚙️ 專案設定載入完成
═══════════════════════════════════════════════════════
  字數要求：{min} - {max}（目標 {target}）
  擴寫範圍：第 {chapter} 章 ~ 第 {chapter+n-1} 章（共 {n} 章）
  處理模式：逐章獨立擴寫
═══════════════════════════════════════════════════════
```

---

### 🔁 以下 Step 1 ~ Step 8 對每一章獨立執行

> [!CAUTION]
> **逐章迴圈規則（強制）**
> ```
> for i = chapter to chapter + n - 1:
>     完整執行 Step 1 ~ Step 8（僅針對第 i 章）
>     ↓ 第 i 章確認完成（字數達標 + ending_summary 已存）
>     ↓ 然後才開始第 i+1 章
> end
> ```
> - **禁止**在第 i 章未完成時就開始考慮第 i+1 章的內容
> - **禁止**同時處理多章
> - **禁止**在完成 n 章之前停下來等待用戶指示
> - 第 i 章完成後，立即自動進入第 i+1 章

---

### Step 1: 讀取章節與字數檢測（第 i 章）
// turbo
讀取 `output/chapters/chapter_{i}.md`，計算中文字數。

```
═══════════════════════════════════════════════════════
  📏 第 {i} 章 — 字數檢測
═══════════════════════════════════════════════════════
  當前字數：{current_words}
  目標範圍：{min} - {max} (目標: {target})
  狀態：{達標 ✅ / 不足 ⚠️}
  需擴充：約 {target - current_words} 字
═══════════════════════════════════════════════════════
```

> [!NOTE]
> 若字數已在 `min` ~ `max` 範圍內，跳過本章，直接進入下一章。

---

### Step 2: 載入上下文（僅為第 i 章服務）
// turbo
**回顧上下文**（確保開頭銜接）：
- 上一章 (i-1) 的 `ending_summary`
- 當前 SubArc 摘要
- 上一章結尾文字（最後 2000 字）

**前瞻上下文**（確保結尾銜接）：
- 讀取下一章 (i+1) 的內容（若存在）

```yaml
前瞻策略:
  下一章為草稿: 讀取完整草稿，提取事件序列前 1-2 項
  下一章為完成正文: 僅讀取開頭 2000 字
  下一章不存在: 無前瞻約束，依 SubArc 大綱自然收束

回顧 + 前瞻 = 本章的「上下文邊界」
  ↳ 開頭必須銜接上一章結尾
  ↳ 結尾必須銜接下一章開頭（若存在）
```

> [!CAUTION]
> **嚴格上下文隔離**
> 僅使用本步驟載入的上下文，不參考對話歷史中的舊章節內容。
> 不預先思考後續章節（i+2, i+3...）的任何內容。

---

### Step 3: 分析內容與擬定擴寫計畫

分析現有內容，判斷擴寫模式：

| 內容類型 | 模式 | 策略 |
|----------|------|------|
| Brief / 骨架 | **全面擴寫** | 將每個事件骨架句轉化為場景，依重要性分配字數 |
| 正文但字數不足 | **補強模式** | 識別薄弱段落，針對性擴充 |

**字數分配規劃（強制）**：
```yaml
目標字數: {target}  # 從 config 讀取，不可自行決定
需補充字數: {target - current_words}

場景分配:
  - 場景 A: ~{x1} 字（描述）
  - 場景 B: ~{x2} 字（描述）
  - 場景 C: ~{x3} 字（描述）
  ...
  總計: ≥ {target}  # 規劃總字數必須 ≥ target
```

> [!IMPORTANT]
> **字數規劃鐵律**
> 1. 規劃階段的總字數分配 **必須 ≥ target**（略多是好的，不夠是不可接受的）
> 2. 不要「先寫再看字數夠不夠」—— 在動筆前就要確保規劃足夠
> 3. 若場景不夠支撐 target 字數 → 在此步驟新增場景或擴充場景細節

```
═══════════════════════════════════════════════════════
  📊 第 {i} 章 — 擴寫規劃
═══════════════════════════════════════════════════════
  模式：{全面擴寫 / 補強}
  目標字數：{target} 字（下限 {min}，上限 {max}）
  當前字數：{current_words} 字
  需補充：約 {delta} 字
  規劃場景數：{scene_count}
  規劃總字數：{planned_total} 字
═══════════════════════════════════════════════════════
```

---

### Step 4: 執行擴寫（分段生成）

> [!CAUTION]
> **分段生成策略（強制）— 這是確保字數達標的關鍵**
>
> **禁止**一次生成整章。必須分段寫作，每段獨立生成後組合。
> 每段寫完後立即計算累計字數，確保朝 target 方向前進。

依 Step 3 的場景分配，**逐場景**生成正文：

```
for each scene in planned_scenes:
    1. 生成本場景正文（依分配的字數目標）
    2. 計算累計字數
    3. 若累計 + 剩餘場景規劃 < target → 擴充本場景或追加場景
end

最終組合所有場景為完整章節
```

> [!TIP]
> **參考分段字數分配**（以 target=4000 為例，可依實際 target 比例調整）：
> - 開場與環境渲染：~800 字（感官描寫、氣氛、銜接前章）
> - 事件發展與互動：~1000 字（角色互動、對話、心理博弈）
> - 高潮與轉折：~800 字（衝突爆發、問題解決）
> - 餘波與結尾：~400 字（情緒沉澱、伏筆、鉤子）
>
> 這不是硬性要求，而是避免「全篇均勻分布」的參考。

擴寫原則：
- ✅ 展開事件的具體過程
- ✅ 深化角色互動與對話
- ✅ 補充與劇情相關的環境描寫、感官描寫
- ✅ 挖掘角色內心活動
- ✅ 遵守 `content_weights` 設定的比例
- ✅ 遵守 `style_profile` 風格設定
- ❌ 堆砌無意義的修飾（灌水）
- ❌ 重複描述同一件事
- ❌ 新增草稿中完全不存在的劇情線

**結尾銜接**：
擴寫最後一個場景時，必須參照前瞻上下文：
1. 本章結尾的角色位置、狀態是否與下一章開頭吻合
2. 本章是否為下一章的觸發事件做好了鋪墊
3. 若有矛盾，調整本章結尾（不可修改下一章內容）

---

### Step 5: 字數校驗（硬性閘門）

> [!CAUTION]
> **字數校驗 — 硬性閘門（HARD GATE）**
>
> 這是不可跳過的強制檢查點。
>
> ```
> final_word_count = count_chinese_chars(expanded_chapter)
>
> if final_word_count < words_per_chapter.min:
>     ❌ 禁止完成
>     ❌ 禁止進入 Step 6
>     → 回到 Step 4，針對字數最少的場景擴充
>     → 可選：追加新場景（環境描寫、心理轉折、角色互動）
>     → 重新通過 Step 5 校驗
>     → 最多重試 3 次，若仍不達標必須報告但不可放棄
>
> if final_word_count > words_per_chapter.max:
>     → 精簡過渡段落和重複描寫
>     → 重新通過 Step 5 校驗
>
> if min <= final_word_count <= max:
>     ✅ 通過，進入 Step 6
> ```
>
> **沒有例外。沒有「字數無法達標就算了」。必須達標。**

---

### Step 6: 一致性驗證

驗證以下項目：
- 擴寫內容不違反已有設定（`lore_bank`、`character_db`）
- 角色行為與性格一致
- 與上一章銜接連貫（回顧上下文）
- 與下一章銜接連貫（前瞻上下文，若存在）

---

### Step 7: 寫入與維護（第 i 章）
// turbo
1. 將擴寫後的完整正文寫入 `output/chapters/chapter_{i}.md`（覆蓋）

2. 輕量維護：

```yaml
必更新:
  lore_bank.yaml:
    - 新增本章事件（若擴寫後有新增細節）
  narrative_progress.yaml:
    - 標記章節完成
    - 更新字數統計
    - completed_chapters[] 新增/更新本章（含 ending_summary）
  character_db.yaml:
    - 更新角色 current_state（若擴寫後有變化）
```

> [!WARNING]
> **ending_summary 為強制欄位**
> 必須生成 `ending_summary`，否則視為章節未完成。
> 此摘要將在下一章的 Step 2 中被讀取。

> [!CAUTION]
> **TOKEN SAVING PROTOCOL**
> 正文直接寫入檔案，對話只輸出路徑、字數、摘要。

---

### Step 8: 本章完成確認
// turbo
```
═══════════════════════════════════════════════════════
  ✅ 第 {i} 章擴寫完成
═══════════════════════════════════════════════════════
  章節標題：{title}
  擴寫前字數：{before_words}
  擴寫後字數：{after_words} / 目標 {min}-{max}（目標 {target}）
  擴展倍率：{ratio}x
  ending_summary：已記錄 ✅
  進度：{completed}/{n}
═══════════════════════════════════════════════════════
```

> [!IMPORTANT]
> **自動繼續（禁止停頓）**
> - 若 `completed < n`：**立即**回到 Step 1 處理第 i+1 章
> - ❌ 禁止詢問用戶「是否繼續」
> - ❌ 禁止停下來等待用戶指示
> - ❌ 禁止報告「已完成第 X 章，請告訴我是否繼續」
> - ✅ 完成第 i 章 → 自動開始第 i+1 章
> - ✅ 完成所有 n 章後 → 進入 Step 9 批次報告

### Step 9: 批次完成報告（所有 n 章完成後）
// turbo
```

  🚀 批次擴寫完成（共 {n} 章）
═══════════════════════════════════════════════════════
  擴寫範圍：第 {chapter} ~ 第 {chapter+n-1} 章

  ├─ 第 {chapter} 章：{before_1} → {after_1} 字（{ratio_1}x）✅

  所有章節達標：✅ / ⚠️
  ending_summary 全部記錄：✅

```

章節已更新至 `output/chapters/`，輕量維護已完成。

## 錯誤處理

| 錯誤類型 | 處理方式 |
|----------|----------|
| 章節檔案不存在 | 停止，提示使用者先執行 /nvDraft 或 /nvChapter |
| 字數校驗失敗（Step 5） | **回到 Step 4 擴充**，不可放棄。最多重試 3 次後報告原因但仍需盡力達標 |
| 角色不一致 | 標記問題並修正後繼續 |
| ending_summary 未生成 | 停止，補生成後再完成 |

---

## 常見陷阱提醒

> [!WARNING]
> **這些錯誤在過去的執行中真實發生過，必須避免：**
>
> 1. **「偷懶」現象**：當 `n` 較大時（如 n=5），LLM 傾向於縮短每章字數。
>    - **解法**：逐章處理，每章都獨立計算字數，不因為「還有很多章要寫」就偷工減料。
>
> 2. **忽略 config 字數要求**：config 寫了 min=3000，結果產出 1500 字就「完成」了。
>    - **解法**：Step 5 的硬性閘門。字數不夠就回去擴充，沒有商量餘地。
>
> 3. **虎頭蛇尾**：第一章寫得很詳細，後面的章節越來越短。
>    - **解法**：每章都是獨立任務，每章都要通過同樣嚴格的 Step 5 校驗。
>
> 4. **未完成就停下**：完成幾章後就停下來詢問用戶是否繼續。
>    - **解法**：Step 8 的自動繼續規則。完成 n 章前不許停。
